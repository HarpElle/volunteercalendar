## 1. What shipped

### Training Sessions — 3 new pages

#### `/dashboard/training-sessions` (admin list)
- Status filter chips: `scheduled` / `completed` / `cancelled` / `all`. Defaults to `scheduled`.
- Each row: title + status badge + date · times · location · ministry · capacity, plus a per-row stat (e.g. "3 accepted · 1 declined" or "5 attended" for completed).
- **+ New Session** opens an inline modal with: title, date, start/end time, location, capacity, prereq dropdown (flattens org-wide + per-ministry prereqs as `Organization · Background Check`, `Worship · Safe Sanctuary Training`), and auto-complete toggle (default ON).
- Empty states differentiate `scheduled` (call-to-action) from other filters (informational).

#### `/dashboard/training-sessions/[id]` (admin detail)
- Session info header with status badge.
- `Send Invitations` button → POST to existing `/invite` handler. Surfaces toast with `"Sent N invitations (M skipped)"` or `"No invitations sent — everyone either already RSVP'd or has the prerequisite cleared."`
- **Roster** lists every eligible volunteer (anyone on the ministry's team, or all active volunteers for org-wide sessions) with their RSVP chip: Accepted / Declined / No response.
- **Mark Complete** flow: clicking flips the roster to attendance-checkbox mode with accepted RSVPs pre-checked; confirm modal explains that the linked prerequisite will be auto-completed for each attendee (or warns if the session has auto-complete OFF). On submit → POST `/complete` → reloads with completed status + "X attendees; Y prerequisite step(s) auto-completed" toast.
- **Cancel session** in a danger-zone box (only visible when status is `scheduled`) → DELETE → returns to list.

#### `/dashboard/training` (volunteer RSVP)
- Reads `?session=X&church=Y` from the email link.
- Resolves the volunteer's Person record via `membership.volunteer_id` (with a Person.user_id fallback).
- Shows session details + my current RSVP (`Going` / `Declined` / `no response yet`) + spots-remaining counter.
- `Accept` / `Decline` buttons call `/rsvp`. Locked out when the session is no longer `scheduled` OR when at capacity AND I'm not already accepted.
- Friendly error states for missing query params, no-volunteer-record, and session-full.

#### Sidebar
New `Training Sessions` entry in the PEOPLE group, right after Onboarding. Gated to scheduler+.

### Release-claim button (bonus)

On `/dashboard/my-schedule` → Upcoming tab, every assignment row is now tagged with `isReleasable: signup_type === "self_signup" AND parent schedule status === "draft"`. For releasable rows, the existing `Can't Make It` / `Remove` buttons are replaced by a single `Release` button:

- Confirm modal asks "Release {role} on {date}?" and explains the slot frees up for another volunteer.
- Click → DELETE `/api/assignments/claim` (existing endpoint from PR #35).
- Optimistic removal from both `assignments` and `allChurchAssignments` state so Upcoming + Open Slots recompute consistently.

Non-releasable rows keep the existing Can't Make It / Remove buttons. The distinction: a `Release` is "I changed my mind before the schedule is locked"; `Can't Make It` is "I had a real conflict on a published schedule" (creates an absence record). Different semantics, different audit trails.

## 8. Edge cases worth watching

- **Org-wide vs ministry-specific prereqs.** The list page detects both. The detail page shows the right roster (ministry team members for ministry prereqs; all volunteers for org-wide). If you want to test the ministry-specific path, add a prereq to a specific team first (e.g. Worship Team → Manage Prerequisites).
- **The volunteer RSVP page is auth-gated by the dashboard layout.** A signed-out user clicking the email link will be sent to login and then redirected back.
- **Sending invitations to volunteers without an email field silently skips them.** The skipped count surfaces in the toast.
- **`Mark Complete` requires `attendee_ids` to be a non-empty array per the API,** but the UI happily lets you submit zero attendees if you uncheck everyone — the confirm modal warns about that case explicitly.
- **Release button on the LAST self-signup on the schedule** correctly frees the slot; the schedule's open-slot count rises by 1. Tested by integration.
- **`Release` is NOT mutually exclusive with the existing `setRemoveItem` flow,** which handles self-removal from published schedules (creates absence records). The conditional render ensures only one shows per row.
