## 1. Root cause (two compounding problems)

### Missing composite index
`GET /api/reservations/requests` runs:

```ts
.where("status", "==", "pending")
.orderBy("created_at", "desc")
```

That combination needs a composite index on `reservation_requests(status ASC, created_at DESC)`. It was never declared in `firestore.indexes.json`. Every queue page load 500'd with `FAILED_PRECONDITION`. The dashboard page silently swallowed the error and rendered the "No pending requests" empty state — same exact pattern as PR #15's missing reservations index.

### PR18 bookings stranded by deploy timing
Your two `PR18 Fellowship...` bookings were created at 22:48:37 and 22:49:22 UTC. PR #18 merged at 22:45:25 UTC. That's ~3-4 minutes after merge — within Vercel's build window. Both bookings hit the **pre-PR-18** POST handler which didn't enqueue approval-required bookings. The PR #18 code itself is correct for new bookings; these two were stranded.

I verified production data directly:
- Fellowship Hall room: `requires_approval: true` ✓
- Both PR18 reservations: `status: pending_approval` ✓
- `reservation_requests` collection had NO docs for them
- Queue GET query failed with FAILED_PRECONDITION

## 9. Edge cases worth watching

- **Index propagation.** I verified the new index finished building via direct Admin SDK query. If you ever see the red error banner again on the queue page, wait a few minutes and reload — Firestore index regions can lag.
- **Recurring approve.** Approve flips every `pending_approval` doc in the group. If you cancelled an occurrence before approving, that one stays cancelled (intentional).
- **Orphan filter.** Queue rows pointing to cancelled/denied reservations are dropped from the response. The underlying request doc still exists in Firestore (status: pending), which is fine — it just doesn't render. Flag if you'd rather it auto-close to `denied`.
- **PR18 bookings on the room detail.** Both reservations show `Pending` badge (Timeline + Reservations tab), inherited from the `status: pending_approval` field. Once you approve/deny via the queue, that badge updates.
- **Future deploy windows.** The backfill script is idempotent and reusable. If we ever ship another POST-side enqueue change and bookings sneak in mid-deploy, the script can run again with no side effects.
