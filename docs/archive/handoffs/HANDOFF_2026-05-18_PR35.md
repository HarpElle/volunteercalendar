# Handoff ‚Äî 2026-05-18 (PR #35: Self-Service open-slot claim flow)

## TL;DR

PR #35 closes the volunteer half of Self-Service mode. Admins could already create a self-service draft (PR #27/#28/#30) but volunteers had no way to claim the open slots. Now they do ‚Äî via a new "Open Slots" tab on `/dashboard/my-schedule`.

This is the **first half of the post-Phase-6 UX sprint**: I built it, automated coverage is green, **Codex needs to validate the actual volunteer experience**. The Codex prompt at the end is what you'd paste to kick off that pass.

## 1. What shipped

### API surface

- **`POST /api/assignments/claim`** ‚Äî race-safe Firestore transaction. Body: `{ church_id, schedule_id, service_id, service_date, role_id, ministry_id }`. Writes an `Assignment` with `signup_type: "self_signup"`, `status: "confirmed"` (the volunteer's click is the confirmation ‚Äî no email round-trip).
- **`DELETE /api/assignments/claim?church_id=‚Ä¶&assignment_id=‚Ä¶`** ‚Äî volunteer releases their own claim, only while the parent schedule is still `draft`.

### UI surface

- **New "Open Slots" tab on `/dashboard/my-schedule`** ‚Äî 4th tab alongside Upcoming / Past / Team. Label shows the count in parens when > 0 (`"Open Slots (3)"`), so the entry point is discoverable without forcing a banner everywhere.
- **Per-row layout** ‚Äî ministry color chip ¬∑ role title ¬∑ date ¬∑ service ¬∑ time ¬∑ `Claim` button.
- **Empty state** ‚Äî friendly copy when no eligible open slots.
- **Inline error** ‚Äî banner above the list for 409s (slot already filled, schedule closed) etc.

### Validation chain (all server-side)

| Code | Reason |
|---|---|
| 401 | Missing bearer token |
| 400 | Missing fields / service_date outside schedule range |
| 403 | Not an active member / no Person record / not on the team / **prereq incomplete or expired** (reuses PR #33's `isJourneyStepEffectivelyValid`) |
| 404 | Schedule or service not found |
| 409 | Workflow ‚âÝ self-service / status not draft\|in_review / slot already filled / already claimed this occurrence |
| 500 | With a "needs composite index" detail if Firestore ever throws `failed-precondition` (PR #32-style canary) |

### Open-slot computation (client-side)

Pure derivation from already-loaded data:
- `selfServiceSchedules` √ó `generateOccurrences` √ó `roles`
- Minus filled (non-trainee) assignments
- Filtered by `canServeInMinistry` + `hasCompletedPrerequisites` (the latter now respects `expires_at` per PR #33)
- Filtered out if the volunteer already has any assignment on the same service occurrence

After a successful claim, the page refetches the all-church assignments slice for that one church so the row disappears from Open Slots + the new assignment shows up on Upcoming on next navigation.

## 8. Edge cases worth watching

- **`generateOccurrences` is imported client-side** for the open-slot computation. Same helper the scheduler uses, so any behavior change there (e.g., the Mon‚ÄìSat √ó Sunday-only case PR #30 covered) automatically applies.
- **Volunteer can claim across multiple churches.** The page already iterates `memberships`, so an admin who joined two orgs sees open slots from both. The Claim handler is per-call, so each row knows its church.
- **Trainee assignments don't count toward capacity.** Same convention as the matrix view (PR #27). If admin marked a slot as Trainee, the slot is still claimable as Regular.
- **The "already claimed same occurrence" guard fires server-side** even if a stale client UI doesn't filter the row out. Defense in depth.
- **No release UI yet.** The `DELETE /api/assignments/claim` endpoint exists + is tested, but the my-schedule UI does NOT yet surface a "Release" button on the Upcoming-tab assignment row. Out of scope for this PR; tracked as a follow-up.

## Open questions / follow-ups

1. **Release-claim UI button on Upcoming-tab rows** ‚Äî currently the only path to release a claim is to call `DELETE /api/assignments/claim` directly. Should be a small "Release this slot" link on self-signup assignments while the schedule is still draft.
2. **Slot-already-filled banner UX** ‚Äî currently surfaces as an inline error above the list. Maybe should also offer "Refresh open slots" so the user can immediately see the latest state without a full reload.
3. **Auto-refresh of the Open Slots tab** when the volunteer navigates back ‚Äî currently relies on the page-load fetch. Mild staleness possible if a slot gets filled in another browser.
4. **Notifications for new open slots** ‚Äî when an admin creates a new self-service draft, should volunteers get a heads-up notification "Open slots posted for Sept"? Could ride on the existing `user_notifications` system added in PR #27.
5. **Sort/filter on the Open Slots tab** ‚Äî currently sorted by date+time. If a volunteer has many eligible slots, ministry filter or grouping might help.

These are all polish; the core flow is in place.
