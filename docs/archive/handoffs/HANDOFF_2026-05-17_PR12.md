## 1. Root cause

The kiosk check-in route resolved a session's room via:
1. `room_overrides?.[childId]` (operator override)
2. `child.default_room_id` (preset on the child record)

If both were missing, `room_id` stayed empty and the session was written as `"Unassigned"`. The Add Child UI doesn't collect `default_room_id`, so freshly-added children **always** landed in Unassigned even when a check-in room had been configured for their grade. The room had the contract (`default_grades`); we just weren't honoring it at check-in time.

This was a missing feature, not a typo or enum mismatch. Stored grades on rooms are canonical lowercase (`"kindergarten"`) and the child's grade is stored the same way, so once you read `default_grades` the match works directly.

## 8. Edge cases worth watching

- **Multiple rooms matching one grade.** When `Room A` and `Room B` both have `Kindergarten` in `default_grades`, we pick alphabetically-first with capacity. That's deterministic but may not match an admin's mental model. Note any cases where you expect a different room.
- **Grade enum / display mismatch.** Storage is canonical lowercase (`kindergarten`); the rooms UI displays capitalized labels (`Kindergarten`). The match runs on the lowercase values, so a visual mismatch shouldn't matter — but flag anything that looks like the UI is showing one value while the kiosk routes to another.
- **Unified vs legacy children.** PR #12 routes both through `assignRoomByGrade`. If you create a legacy-style child elsewhere and the lookup behaves differently, that's worth flagging.
- **Checked-out sessions disappearing from teacher view.** The teacher room view filters out checked-out sessions (`checked_out_at` is non-null hides them). That's intentional — the teacher view is for "who's in my room right now." If you expect to see historical sessions there, that's a separate request.
- **Capacity overflow.** The existing capacity-SMS + overflow_room redirect logic runs on the room we matched. If the matched room is full and has an `overflow_room_id`, the check-in lands in the overflow room and the SMS fires.

## Open questions / follow-ups (out of scope for this PR)

1. Should `Add Child` collect `default_room_id` so an admin can override the grade match per child? Probably yes for siblings who should always go to the same room, but it's a UI add.
2. The kiosk currently picks rooms purely by alphabetic order + capacity. Should we expose a UI-controlled priority/sort field (`order`, `priority`)? Not blocking — flag if you see a case where the picked room feels wrong.
3. SMS pre-checkin path probably also needs the unified treatment but Codex hasn't tested it yet.
