## 1. Root causes

| # | Blocker | Root cause |
|---|---------|------------|
| 1 | People archive no-op | `handleArchiveVolunteer` (and `handleRemoveFromOrg`) in `people/page.tsx` each called `window.confirm()` AFTER `person-detail-drawer.tsx` already showed its styled confirm modal. Browser automation auto-dismissed the native dialog (returning `false`), so the handler bailed out before the PATCH request fired. Users with non-headless browsers likely cancelled silently or saw a second prompt that felt redundant. |
| 2 | Training Sessions UI missing | API routes + email template + auto-complete cron were shipped (Phase 30-ish), but no admin-facing page was ever built. CLAUDE.md overstated the feature's status. |
| 3 | Prerequisite 365-day expiration not exposed | `OnboardingStep` type had no `expires_in_days` field at all; the editor only knew about `type` / `label` / `threshold` / `scope`. The TEST_PLAN promised a feature that didn't exist in the schema. |
| 4 | Prerequisite completion email never arrived | `updateJourneyStep` in `onboarding/page.tsx` wrote the journey doc but never called `/api/notify/prerequisite`. The endpoint, the email template, and the in-app notification helper were all already wired — just no caller. |
| 5 | Short Links has no create/expiry UI | The page was designed around share-button-generated links only; POST validates `expires_in_days` between 1 and 90 days, so testers couldn't directly create an already-expired link to verify the `/s/<slug>` 404 path. |
| 6 | Invite accept "Failed to load invitation" then "You've joined ." | Page read membership + church doc directly from Firestore on the client. Cross-org invitees aren't members of the destination church yet, so the church-doc rule check fails. The catch block set the error banner but left `status` as `"idle"`, then accept proceeded with an empty `churchName`. |
| 7 | Phase 6 multi-org setup mismatch | TEST_PLAN started Phase 6 from a state where one account was a member of two orgs, but the rest of the document never described how to get there. |

## 8. Edge cases worth watching

- **Existing prerequisites have no `expires_in_days`.** They continue to write `expires_at: null` on completion, same as before — no migration needed. Newly-edited prereqs with a value start computing expiry.
- **Short-link POST still caps at 90 days expiry.** Testers can't directly create a *past* expiry. The "Expire now" PATCH is the supported path. POST with negative or zero gets coerced to the default.
- **Invite-accept "wrong_user" branch** kicks in when the signed-in UID doesn't match `membership.user_id`. Same UX as before; just now driven by the API's 403 response instead of a client-side check.
- **Prerequisite-completion email is fire-and-forget.** If Resend is down or the call 500s, the journey write still lands. Worst case the volunteer doesn't get the email; the in-app notification is also best-effort but written via admin SDK so it's much less likely to fail.
- **`getAuth().currentUser` inside `updateJourneyStep`** — this is the Firebase client SDK lookup; should be safe inside this React handler since the user is authenticated to reach the page.
