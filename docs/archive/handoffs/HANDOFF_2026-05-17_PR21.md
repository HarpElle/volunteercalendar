## 1. Root cause

`GET /api/display/room/[roomId]` runs:

```ts
.where("room_id", "==", id)
.where("date", "==", today)
.where("status", "==", "confirmed")
.orderBy("start_time", "asc")
```

Firestore requires a composite index on `reservations(room_id ASC, date ASC, status ASC, start_time ASC)`. It was never declared. Every wall-display GET returned 500 with the raw FAILED_PRECONDITION text including the console URL. Same class as PR #15 (reservations list) and PR #19 (approval queue) — a missing index that silently broke a feature until a real user hit the page.

## 2. Fixes

- **`firestore.indexes.json`**: added the composite index. Deployed via `firebase deploy --only firestore:indexes`. **Index has finished building** — verified by running the API-equivalent query through the Admin SDK against production for the Sanctuary; succeeds with 0 rows (today is not Sunday, no Sanctuary booking).
- **`/api/display/room/[roomId]`**: catches `FAILED_PRECONDITION` specifically and returns a friendlier message ("Display data is initializing. Please try again in a minute.") instead of the raw Firestore string. Wall-mounted displays should never expose Firestore console URLs.

## 8. Denied visibility decision

**Intentional.** `/api/reservations` default GET filter is `confirmed + pending_approval`. Denied bookings drop out of the room detail's Timeline + Reservations tabs because they no longer claim the slot — same reason cancelled bookings disappear.

- Logical position: the room detail is the "what's actually booked here" view. Denied = not booked. Audit trail (who approved/denied what, when) belongs in a future history view.
- I left the `Denied` badge in the Timeline render code from PR #16 — it's dead today but will render correctly if a future change adds a "Show denied" toggle. Cheap insurance.

If you'd prefer denied bookings to be visible-with-Denied-badge, flag it and I'll either:
(a) Update the room detail GET to include denied, or
(b) Add an explicit "Show denied" toggle in the date range chip row.

For now: update test plan expectations to "denied reservations disappear from the room detail by design."

## 10. Edge cases worth watching

- **Index propagation.** Verified built via Admin SDK. If you see the FAILED_PRECONDITION error again somehow, wait a minute and reload — Firestore index regions can lag.
- **Wake lock.** The display page calls `useWakeLock()` to keep the screen on. On some browsers / OS combos this needs user interaction first. Flag if the display dims unexpectedly after a few minutes.
- **30s poll.** The display polls `/api/display/room/[roomId]` every 30 seconds. New reservations show up within that window. The clock updates every 1 second.
- **`status === "confirmed"` filter.** The display only shows confirmed bookings — `pending_approval` is hidden. So if you're testing Fellowship Hall (which requires approval), the bookings won't show on its wall display until you approve them via the queue. Intentional.
