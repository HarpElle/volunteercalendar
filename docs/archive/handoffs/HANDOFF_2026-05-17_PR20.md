## 1. Root cause

**Anonymous queue cards.** The page read `req.reservation.title`, `req.reservation.requested_by_name`, etc. But `/api/reservations/requests` returned `requests[]` and `reservations[]` as parallel arrays — the join was never done client-side or server-side. `req.reservation` was always undefined. Every card fell back to placeholder text (`"Reservation"`, `"Unknown"`).

**Immediate Deny.** Clicking Deny fired the API call instantly. Combined with anonymous cards, a misclick was indistinguishable from intent and irreversible from the UI. Your automation accidentally denied `PR17 Conflict Override` during the queue smoke test for exactly this reason.

## 2. Fixes

### API (`GET /api/reservations/requests`)
- Embeds the full `reservation` shape on each request: title, room_id, room_name, date, start_time, end_time, requested_by_name, is_recurring, recurrence_group_id, attendee_count, equipment_requested, description, setup_notes, status.
- Resolves and embeds `room_name` (batched lookup across all rooms referenced).
- Resolves every `conflicting_reservation_id` and embeds the full `conflicts` array (each: id, title, date, start_time, end_time, requested_by_name).
- New top-level `reason` field: `"conflict"` (booking overlapped) or `"approval_required"` (room/org required approval). Derived from stored `reason` (PR #18) or inferred from conflicts when missing.
- Defensive orphan filter (PR #19) still drops queue rows whose target reservation is cancelled/denied. Now happens in the same pass.
- Keeps the parallel `reservations` array for back-compat with older clients.

### Queue page (`/dashboard/rooms/requests`)
- Renders embedded data directly. Each card now shows:
  - Title
  - `Recurring` badge (for recurring requests)
  - `Conflict` or `Approval required` reason badge
  - "Requested by [name]"
  - Two-column grid: Room, Date, Time, Series note (recurring), Attendees, Equipment, Description, Setup notes
  - Inline conflict list (title + time + requester for each conflicting reservation)
- Approve and Deny now open a shared `ConfirmModal` showing the same details. Modal has:
  - Note field (recommended for Deny, optional for Approve)
  - Amber warning on Deny explaining series-wide impact for recurring
  - Explicit Cancel + Confirm buttons
- Errors during approve/deny surface in the modal banner instead of swallow.

## 9. Edge cases worth watching

- **Recurring approve fans out.** Approving a recurring request flips every `pending_approval` doc in the recurrence group. Cancelled/denied occurrences are left alone (deliberate — don't resurrect intentional cancellations).
- **Recurring deny semantics.** Deny does the same — flips every `pending_approval` doc in the group to `denied`. If you only want to deny one occurrence, cancel that occurrence individually first (via the room detail Cancel button), then approve the rest from the queue.
- **Note recommended on Deny.** The modal says "recommended" not "required" — the underlying DELETE-like endpoint still accepts an empty note. Flag if you want it required.
- **Approve modal.** Approve also opens a confirmation modal even though it's non-destructive. If that feels excessive for high-volume queues, flag and I'll switch Approve back to a one-click action.
- **Orphan filter still active.** Any request whose target reservation has been cancelled/denied since the request was created is hidden from the UI. The request docs still exist in Firestore.
- **Accidental PR17 deny.** Left as-is for now. Flag if you'd like it restored.
