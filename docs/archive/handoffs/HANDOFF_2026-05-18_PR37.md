## 1. Root cause

### Issue #1 (HIGH) — Claimed draft/in-review assignments invisible in Upcoming

The volunteer's `/dashboard/my-schedule` page and the `/api/calendar` iCal feed both applied a `schedule.status === "published"` filter to every assignment. That filter dates back to PR #27 (Codex QA 2026-05-15), which closed a real bug: admin-pushed draft assignments were leaking into volunteers' views before the schedule was approved.

PR #35 then introduced a *different* class of draft-state assignment — the volunteer's own **self-signup** claim — and the PR #27 filter happily hid it too. So the volunteer clicked "Sign Up", got a 200, the row vanished from Open Slots... and then there was no visible record of it anywhere until the schedule eventually got published. Codex correctly flagged this as a trust problem.

### Issue #2 (MEDIUM) — 409 race left stale row visible

`handleClaim` only refetched the all-church assignments slice on success. The 409 path returned early with an error banner but no state refresh, so the open-slot computation still believed the slot was unfilled and kept the stale row in the list.

## 2. Fix

### Self-signup carve-out

**The carve-out only widens THE CLAIMANT'S view of their OWN opt-in records.** Cross-volunteer privacy is preserved.

- **`src/app/dashboard/my-schedule/page.tsx`** — page-load filter now accepts assignments where `signup_type === "self_signup"` (in addition to `schedule.status === "published"`). The fetch query already restricts to `person_id === me`, so the filter relaxation is necessarily scoped to the volunteer's own data.
- **`src/app/api/calendar/route.ts`** — same carve-out, scoped to **PERSONAL feeds where `person_id === target_id`**. Team / ministry / org feeds keep the published-only rule (they're the "official lineup" view — a single volunteer's draft opt-in shouldn't surface to the whole team via someone's team .ics).

### 409 recovery

- `handleClaim` now refetches the church's assignments **on both success and 409**. Also refetches the volunteer's own assignment slice so a successful claim is immediately visible on **Upcoming** without a page reload.
- The 409 message now reads: `"That slot was just filled. We refreshed the list."` — matches Codex's recommended copy.

### UX polish bundled (per Codex feedback)

- Button label: `"Claim"` → **`"Sign Up"`**
- In-flight label: `"Claiming…"` → **`"Signing up…"`**
- Success toast: **`"Signed up for {role} on {date}. See it on Upcoming."`** Green banner that auto-dismisses after 5s.
- Tab name unchanged (`"Open Slots"` — Codex OK'd it).

## 9. Edge cases worth watching

- **Existing data updates retroactively.** Any self-signup claim that landed before PR #37 (Codex's PR #36 retest data) becomes immediately visible to its claimant after deploy. No backfill needed; it's a render-time filter change.
- **Schedule transitions still work.** A self-signup claim on a draft schedule shows in Upcoming. When the admin moves the schedule to in_review → approved → published, the assignment stays in Upcoming with no visible change (the carve-out covers the in_review state too; once published, the regular filter accepts it anyway).
- **Releasing a claim** still has no UI button — Codex's known gap from the PR #35 handoff. A volunteer who self-signed up can't undo from the my-schedule UI; the `DELETE /api/assignments/claim` endpoint exists if needed. Tracked as a follow-up.
- **Team/ministry/org iCal feeds** intentionally keep the published-only rule. If a tester wants to see a draft self-signup on a non-personal feed, that's by-design absent. The "official lineup" view shouldn't surface single-volunteer opt-ins.
- **The 409 toast says "We refreshed the list."** but the refetch happens in parallel with rendering the toast. If the refetch fails (network), the toast appears but the row stays. Best-effort; degraded behavior is the same as before this PR.
