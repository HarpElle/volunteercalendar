## PR #33 — Scheduler `expires_at` gating

### Root cause

PR #31 added `OnboardingStep.expires_in_days` and started writing `VolunteerJourneyStep.expires_at` on completion (365 days for a typical background check). But the scheduler (`hasCompletedPrerequisites`) and the onboarding pipeline (`getOrgEligibility` / `getVolunteerStage`) both read only `status === "completed"|"waived"` and ignored `expires_at`. So a volunteer whose background check ran out yesterday would still be auto-assigned to children's-ministry roles today — exactly the failure mode the expiration system is supposed to prevent.

### Fix

New `isJourneyStepEffectivelyValid(step, todayIso)` helper in `src/lib/utils/eligibility.ts`. Returns `true` iff `status` is `completed`/`waived` AND `expires_at` (if set) is on a future calendar day. Date-string comparison so a 3 a.m. expiry doesn't accidentally read as "still valid" later that day.

- `hasCompletedPrerequisites` in `scheduler.ts` calls it for org-wide + ministry-specific prereqs. Expired step → volunteer skipped → slot becomes unfilled-role conflict.
- `getOrgEligibility` + `getVolunteerStage` demote a previously-cleared volunteer with an expired step from `cleared` → `in_progress` (NOT `not_started` — the admin needs to know they did the prereq before and need to renew it).

### Tests

- `tests/unit/journey-step-expiry.test.ts` — 13 cases on the helper + the two eligibility entry points.
- `tests/unit/scheduler-expired-prerequisite.test.ts` — 4 end-to-end cases: expired = skipped, valid = assigned, scheduler prefers valid over expired, backward-compat with no expires_at.

152 unit tests pass on `main`.

### Files changed

- `src/lib/utils/eligibility.ts` — new helper + transition logic
- `src/lib/services/scheduler.ts` — use helper in prereq check
- `tests/unit/journey-step-expiry.test.ts` (new)
- `tests/unit/scheduler-expired-prerequisite.test.ts` (new)

### Firestore / data

**None.** No rules, indexes, schema, migrations, or data changes. Existing data with no `expires_at` continues to behave the same way.

## PR #34 — CI smoke for write-API endpoints

### Root cause

PR #32 was a missing-composite-index bug on `POST /api/short-links`. The page surfaced `{"error":"Internal error"}` with a 500; the actual `FAILED_PRECONDITION` from Firestore landed in Vercel logs but never in the HTTP response. Codex hit it on production because we had no pre-deploy coverage of the POST query shape.

### Fix

Emulator-backed integration tests for write-API route handlers. The emulator IS index-strict in the same way prod is, so a missing index throws `FAILED_PRECONDITION` in CI exactly like it would in Vercel.

**Pattern (small enough to apply per-endpoint):**
1. `vi.mock("@/lib/firebase/admin", ...)` at the top of the test file — `adminDb` points at emulator; `adminAuth.verifyIdToken` stub treats the Bearer token as the UID.
2. `beforeEach` calls `resetFirestore` + `seedChurchAndMemberships` from `tests/integration/_seed.ts` (clears top-level + per-church subcollections; seeds owner/admin/scheduler/volunteer memberships at known UIDs).
3. Invoke the route handler directly with a `NextRequest`; assert response status + Firestore state.

**Coverage in this PR (22 tests across 2 files):**
- `tests/integration/short-links.test.ts` (14 tests) — POST happy path (the PR #32 canary) + all 4xx branches; PATCH `expire_now`; DELETE; GET.
- `tests/integration/assignments-patch.test.ts` (8 tests) — PR #27's trainee toggle. Proves the pattern handles dynamic route params + sub-collection paths + scheduler-OR-admin role gate.

### CI changes

- `.github/workflows/ci.yml` — new step `API integration tests (emulator)` after the existing Firestore rules tests step. Same Java + Firebase CLI setup; reuses the existing infrastructure.
- New npm scripts: `test:integration` (just integration tests), `test:emulator` (rules + integration together). Both pass `--no-file-parallelism` because the emulator is a single shared instance and parallel `resetFirestore` runs would corrupt each other.
