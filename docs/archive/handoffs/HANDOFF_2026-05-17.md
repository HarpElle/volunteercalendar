### Worship (3 blockers + 2 polish)
- **`Add Song`** button on `/dashboard/worship/songs` now opens a Create modal (Title + optional Default Key). Submits to `POST /api/songs`, navigates to the editor.
- **`Edit`** button on each song row now navigates to `/dashboard/worship/songs/[id]`.
- **`New Plan`** button on `/dashboard/worship/plans` now opens a Service + Date picker, POSTs to `/api/service-plans`, navigates to the plan editor. The button is disabled if there are no Services yet (clear empty-state hint).
- **Archived tab**: each archived song now shows a green **Restore** button that calls `PATCH /api/songs/[id]` with `status: "active"`.
- **`Archive`** button now actually persists (PATCH) — previously it only updated local state, which is why archive didn't survive a reload.
- **Songselect import duplicate detection** now runs during preview, not after `Import All`. Duplicates show an `Already in library` badge and a disabled `Skipped` button. The progress summary now reads "1 new song ready to import · 1 already in your library (will be skipped)" instead of the ambiguous "1 imported, 1 already in library" message.

### Rooms (1 blocker)
- **Create silent failure root cause**: the GET query combined `where("is_active") + orderBy("name")` without the matching Firestore composite index, so the listing endpoint 500'd after the create succeeded. The room was getting created but the refetch returned nothing.
- **Fix**: dropped the `orderBy` from the query and sort client-side. Also added the composite index for future-proofing and deployed it.
- Errors on load and create now display in the UI (no more silent fail).
- Admin view now requests `include_inactive=true`, so newly-created rooms appear immediately even if you toggle them off.

### Children's Check-In (4 blockers + 1 high)
Root cause for issues #7, #8, #11, #12 was the same: **unified vs legacy collection drift**. The Pro-tier checkin household POST already wrote to the unified `households` + `people` collections, but the GET/PUT/DELETE, regenerate-qr, kiosk register, and CSV import endpoints all still read/wrote the legacy `checkin_households` + `children` collections. Households were being created somewhere the rest of the stack couldn't see.

- **Household detail "not found"**: `/api/admin/checkin/household/[householdId]` GET now detects unified mode and reads from `households` + `people`, mapping back into the legacy `CheckInHousehold` + `Child` shape the admin UI expects. PUT updates the linked adult Person docs. DELETE cascades through everyone linked to the household. Regenerate-QR writes to whichever collection holds the household.
- **First-time visitor 500**: `/api/checkin/register` was unconditionally writing `secondary_guardian_name: undefined` when the registrant skipped the secondary guardian field. Firestore rejects `undefined`, which surfaced as 500. Fixed by building the payload conditionally. Also added unified-mode writes so registered families show up in lookup and the admin household list.
- **Grade enum mismatch**: visitor-registration emitted `pre_k` but the Child schema requires `pre-k`. Added `toddler` to the picker and added a server-side normalizer that maps `pre_k → pre-k` and rejects unknown values.
- **CSV import false success**: now writes to unified collections when the church has unified `people`, so imported families appear in the household list. Also fixed the row counter (1-row CSV with trailing newline used to report "2 data rows" — now correctly reports "1").

### Dashboard Kiosk Setup (1 high)
- The dropdown previously displayed the church's 6-letter `short_code` as a "Setup Code". The kiosk explicitly rejects anything that isn't `^[0-9A-F]{8}$`. The short_code is a church identifier, not a kiosk credential — it never worked, regardless of the code value.
- **Fix**: replaced the panel with a `Generate Activation Code` button that calls `POST /api/admin/kiosk/stations` and shows the valid 8-character hex code. This produces the same code the Stations tab does, so both entry points now work.

## Known intentional decisions / not bugs

- **`Add Song` is a stub-then-edit flow**, not a multi-step wizard. We deliberately keep it minimal because Songselect import is the canonical bulk path; manual add covers public-domain or one-off songs.
- **`New Plan` requires an existing Service**. If none exists, the button shows a link to `/dashboard/services-events`. This is intentional — service plans are scoped to a specific service+date.
- **Kiosk Setup activation code is per-station**, not per-church. Generating a new code creates a new Station. Manage all stations under Settings → Stations.
