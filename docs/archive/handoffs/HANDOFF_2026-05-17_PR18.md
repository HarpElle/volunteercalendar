## 1. Root cause

`POST /api/reservations` created the `reservation_requests` queue doc only inside an `if (status === "pending_approval" && hasConflict)` guard. So approval-required bookings that had no overlap got `status: "pending_approval"` on the reservation itself but no queue doc, and the requests page (which reads from `reservation_requests`) saw nothing.

Recurring approval-required bookings had the same gap: the recurring branch of POST never wrote a request doc at all.

The conflict-override case worked because it explicitly wrote a request doc. Pure approval-required cases (Fellowship Hall's `requires_approval`) didn't.

## 2. Fixes

### POST single reservation
- Queue doc is now written whenever `status === "pending_approval"`, not just on conflict.
- Doc carries a new `reason: "conflict" | "approval_required"` field so the queue UI can render context later.

### POST recurring reservation
- After materializing the occurrences, when `status === "pending_approval"` the handler writes **one** queue doc with `recurrence_group_id` set. The admin approves/denies once for the whole 12-week series instead of 12 individual requests.

### Approve / Deny endpoints
- When the request carries `recurrence_group_id`, batch-update every `pending_approval` doc in the group at once. Skips `cancelled` / `denied` siblings so a partially-cancelled series doesn't get resurrected.
- Single bookings keep today's one-doc path.

### Type
- `ReservationRequest` gained `reason?: "conflict" | "approval_required"` and `recurrence_group_id?: string`.

### UI
- Room detail Timeline rows now show a `Pending` badge (warning) for `pending_approval` and `Denied` (danger) for `denied`. Reservations tab already had badges; Timeline now matches.

## 8. Edge cases worth watching

- **`reason` field.** New queue docs carry `reason: "conflict" | "approval_required"`. Old queue docs (pre-PR-18) don't have this field. The queue UI doesn't currently render it, so this is forward-looking — but flag if the queue treats them differently than expected.
- **Recurring approve.** When you approve a recurring booking, every `pending_approval` occurrence flips to `confirmed`. `cancelled` and `denied` siblings are NOT touched — so if you cancelled a single occurrence before approving, that one stays cancelled.
- **Volunteer-submitted bookings.** A volunteer (Alex) can submit a booking; it follows the same `pending_approval` → queue → approve/deny flow. The SMS notification on approve/deny is sent to the requester's phone if a membership phone is on file.
- **Conflict override + approval-required combined.** If you book a Fellowship Hall slot that ALSO conflicts with another reservation, the queue doc records `reason: "conflict"` (conflict wins as the diagnostic label). The reservation is still `pending_approval` and approve/deny works the same way.
- **Edit-to-overlap.** PUT edit-to-overlap still returns 409 with the conflict modal — that path is unchanged by this PR. It does NOT re-enter the queue; the user is forced to pick a different time or explicitly override.
