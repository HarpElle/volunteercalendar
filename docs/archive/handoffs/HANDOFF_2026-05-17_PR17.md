## 1. Root cause

`POST /api/reservations` did detect the overlap — the `findConflictsInTransaction` helper found the existing Bible Study reservation. But instead of returning 409, the transaction wrote the new doc with `status: "pending_approval"` and `conflict_with_ids: [...]`, then returned 201.

The booking form's `ReservationConflictModal` only fires on `!res.ok && json.conflicts`. A 201 response is `res.ok = true`, so the modal never appeared. The user saw "booking accepted" and a `pending_approval` reservation silently persisted.

Compounding factors:
- The `allow_conflict` field the form already sent was never read on the server.
- The PUT endpoint had no conflict re-check, so edit-to-overlap also slipped through silently.
- `findConflicts*` helpers returned only IDs, so even if 409 had fired, the modal couldn't have rendered titles/times.

## 2. Fixes

### POST single reservation
- Read `allow_conflict` from body (default `false`).
- Transaction now returns a typed `{ kind: "conflict", conflicts }` when conflicts exist AND `!allow_conflict`. Handler responds with **HTTP 409** + `{ error, conflicts: ConflictDetail[] }` and **does not write** the doc.
- `allow_conflict: true` (the override retry the form sends from the modal) keeps today's behavior: write with `pending_approval` and `conflict_with_ids` populated.

### POST recurring reservation
- Mirrors single behavior. Any overlap across any generated occurrence date → 409 with the full list of conflicts unless `allow_conflict=true`.

### PUT (edit time)
- New edit-to-overlap check. If `start_time` or `end_time` is changing, re-check every reservation the edit will touch:
  - `single_date` / non-recurring → just the one doc
  - `from_date` → group siblings with `date >= existing.date`
  - `all` → every doc in the recurrence group
- Excludes self + siblings from the conflict set so editing a recurring series doesn't conflict with its own other occurrences.
- Returns **409** with `{ error, conflicts }` unless `allow_conflict=true`.

### Helpers
- Exported `intervalsOverlap(aStart, aEnd, bStart, bEnd)` — open-interval check: `aStart < bEnd && bStart < aEnd`. Back-to-back bookings (one ends exactly when the next starts) do **not** conflict — intentional, supports setup/teardown chains.
- `findConflicts*` now return rich `ConflictDetail` records (id, title, date, start_time, end_time, requested_by_name) so the modal can show the conflicting booking.
- Cancelled + denied reservations don't claim the slot; `pending_approval` still does.

## 9. Edge cases worth watching

- **Open-interval semantics.** Back-to-back bookings (10:00–11:00 + 11:00–12:00) do not conflict. This is the standard convention for scheduling and supports immediate setup/teardown chains. Flag if any user expects an inclusive overlap.
- **Pending approval still claims the slot.** A `pending_approval` booking blocks new conflicting bookings just like `confirmed` does. Only `cancelled` and `denied` are non-blocking. Flag if you expected to be able to "compete" for a pending slot.
- **Override path.** The modal's "Override" button retries with `allow_conflict: true`. The new booking lands in `pending_approval` with `conflict_with_ids` populated. There's no UI yet to inspect what was conflicted with after the fact other than the `pending_approval` status.
- **Edit-to-overlap with `from_date` scope.** The check considers ALL siblings from that date forward. So editing time on a Tuesday-Wednesday-Thursday recurring series with scope `from_date` correctly checks each subsequent Tuesday, Wednesday, Thursday for non-sibling conflicts.
- **Self-exclusion.** A reservation cannot conflict with itself or with its own siblings when editing a series. Verified via the `targetIds` Set in the PUT handler.
