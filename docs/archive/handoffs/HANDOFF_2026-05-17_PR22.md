## 1. Root cause

Multiple sites computed "today" via `new Date().toISOString().split("T")[0]` — that's **UTC**. After ~7pm America/Chicago, the UTC date is already on the next day, so every query for "today" silently missed your same-day evening reservation:

- `/api/display/room/[roomId]` queried 2026-05-18 instead of 2026-05-17 → display showed empty
- `/api/reservations` GET default `date_from` was 2026-05-18 → room detail Timeline + Reservations both hid the booking
- Room detail page client computed `today = 2026-05-18` and shipped it as `date_from` → would have still missed it even with a partial server fix

## 2. Fixes

- **`src/lib/utils/date.ts`** — new `todayInTimezone(tz)` helper using `Intl.DateTimeFormat`. Returns YYYY-MM-DD in the church's local calendar day. Falls back to UTC for invalid/missing tz. Works in both Node (server) and browser (client).
- **`/api/display/room/[roomId]`** — reads `church.timezone`, uses `todayInTimezone(tz)` for the today filter.
- **`/api/reservations` GET** — default `date_from` resolves to church-local today. Explicit `date_from` callers unaffected.
- **`/dashboard/rooms/[roomId]` page** — pulls `church.timezone` alongside `subscription_tier` in the existing Firestore fetch. `fetchReservations` computes `date_from` and `date_to` in church TZ before requesting.
- **`/api/calendar/room/[roomId]/[calendarToken]`** — iCal feed window (-30 days … +60 days) now anchored on church-local today.

## 10. Edge cases worth watching

- **Wall display 30-second poll.** Same as PR #21. Booking → 30s later the display refreshes.
- **Server fetches church doc on every GET that defaults `date_from`.** One extra read per request. Acceptable; can be optimized with a cache later.
- **DST transitions.** `Intl.DateTimeFormat` handles DST correctly. If you're testing during a DST switch (Mar 8, Nov 1 in 2026), the "today" boundary may shift an hour but the date string remains the local calendar day.
- **Client falls back to "UTC" if the timezone fetch fails.** If the church doc somehow doesn't load, the client will revert to UTC behavior. Loud failure would be better; today it's silent. Flag if it bites.
