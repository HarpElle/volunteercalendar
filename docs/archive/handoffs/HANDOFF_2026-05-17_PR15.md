## 1. Root cause

`GET /api/reservations` runs:

```ts
adminDb.collection(`churches/${cid}/reservations`)
  .where("date", ">=", dateFrom)
  .orderBy("date")
  .orderBy("start_time")
```

Firestore requires a composite index on `(date ASC, start_time ASC)` for that combination. The index was never declared in `firestore.indexes.json`, so the query returned `9 FAILED_PRECONDITION: The query requires an index` (HTTP 500). The room detail page's `fetchReservations` silently swallowed the error, so the failure surfaced as "No upcoming reservations" rather than "fetch failed."

This was a pure infra gap. The recurrence wizard, the POST transaction, the recurrence rule, the materialization of 12 docs — all of those worked correctly. The page just couldn't read them back.

## 2. Fix

1. Added composite index `reservations(date ASC, start_time ASC)` to [`firestore.indexes.json`](firestore.indexes.json). Deployed with `firebase deploy --only firestore:indexes`. **Index has finished building** — verified by running the API-equivalent query directly through the Admin SDK against production data.
2. Room detail page's `fetchReservations` now surfaces the API error as a red banner under the tab bar instead of silently swallowing it. Future missing-index conditions (or any other 500) will show the actual error text.

No application code change to the query itself — it was correct, just unindexed.
