## 1. Root cause

Two failures, same underlying cause.

### 1a. TypeScript bug introduced in PR #35

`src/app/dashboard/my-schedule/page.tsx:568` assigned `occ.service.end_time` (`string | null` per `Service.end_time`) into `OpenSlot.endTime` which I'd typed as plain `string`. Vercel's `next build` is strict about this and bailed out:

```
./src/app/dashboard/my-schedule/page.tsx:568:15
Type error: Type 'string | null' is not assignable to type 'string'.

    serviceDate: occ.date,
    startTime: occ.service.start_time,
>   endTime: occ.service.end_time,
```

### 1b. Process gap

GitHub Actions caught the same error on the `pr35-self-service-claim` branch (run `26041573306` → failure on "TypeScript typecheck"), but I squash-merged the PR without checking the CI status. The post-merge CI run on `main` also went red, and Vercel's production build silently failed — so production kept serving the previous green deploy (no Open Slots tab anywhere).

Confirmed by checking the Vercel deploys list: every build after PR #35 errored until PR #36 fixed the type.

## 2. Fix

- **`src/app/dashboard/my-schedule/page.tsx`** — `OpenSlot.endTime: string | null` to match `Service.end_time`. The render block already uses `?:` to render an `–` separator only when both times exist, so no JSX change needed.
- **`.github/workflows/ci.yml`** — new step **"Smoke check — critical UI strings present in build output"** after the production build. Greps `.next/static` for user-visible strings that must survive into the built bundle. Initial three pins:
  - `"Open Slots"` (PR #35 tab)
  - `"Send Availability Request"` (PR #27 button)
  - `"Create a test link"` (PR #31 short-links admin form)
- Adding new pins is ~1 line per surface we care about regression-protecting.
