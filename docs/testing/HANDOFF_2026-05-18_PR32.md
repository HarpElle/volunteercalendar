# Tester Handoff — 2026-05-18 (PR #32: short-links POST composite index)

## TL;DR for Codex

Short Links `Create test link` now works on production. The 500 was a missing Firestore composite index; it's now deployed and READY. **Stay in `TESTER — Codex 2`. After the Short Links slice + a smoke sweep, Phase 6 is done.**

PR: [#32](https://github.com/HarpElle/volunteercalendar/pull/32), squash-merged at commit `4f4578e`.

## 1. Root cause

`POST /api/short-links` runs the tier-limit query:

```ts
adminDb.collection("short_links")
  .where("church_id", "==", church_id)
  .where("expires_at", ">", now)
  .get();
```

Equality + range on two different fields requires a composite index. We had `short_links/{church_id ASC, created_at DESC}` (powers the GET ordering) and `short_links/{slug ASC, expires_at ASC}` (powers slug uniqueness), but NOT `{church_id ASC, expires_at ASC}`. Firestore threw `FAILED_PRECONDITION`; the bare catch returned `{"error":"Internal error"}` with a 500 and no breadcrumb in the response.

It worked for me when I tested in PR #31 only because... actually I didn't manually exercise the POST against production after merge — that's the gap. The path was technically broken on day one of PR #31; Codex's run is the first real POST attempt.

## 2. Fix (two parts)

1. **`firestore.indexes.json`** — added the missing composite index:
   ```json
   {
     "collectionGroup": "short_links",
     "queryScope": "COLLECTION_GROUP",
     "fields": [
       { "fieldPath": "church_id", "order": "ASCENDING" },
       { "fieldPath": "expires_at", "order": "ASCENDING" }
     ]
   }
   ```
   Also created directly on production via Firebase MCP so it didn't have to wait on a `firebase deploy` cycle. Index name: `projects/volunteercalendar-mvp/databases/(default)/collectionGroups/short_links/indexes/CICAgNja0ogK`. State: **READY** as of this handoff.

2. **`src/app/api/short-links/route.ts`** — POST catch now detects `failed-precondition` (string code or gRPC 9) and returns:
   ```
   { error: "Short-link query needs a composite index. Re-run `firebase deploy --only firestore:indexes` and try again.",
     detail: <original error message — typically includes the create-index URL> }
   ```
   Other errors keep the generic top-level message but now also include `detail`. Net effect: the next missing-index-style 500 will land as a self-explanatory hint in the HTTP response, not just in Vercel logs.

## 9. Edge cases worth watching

- **The PATCH "Expire now" path uses just a `doc.get()` + `doc.update()`** — no composite query — so it was never affected by the missing index. Codex couldn't reach it because the create POST failed first; now that POST works, PATCH should work first try (it was already merged in PR #31).
- **GET `/api/short-links?church_id=…`** uses `church_id + created_at`, which was already indexed; that's why Codex saw `200 {"links":[]}` from the initial list. Net effect for testers: a healthy GET is no longer a guarantee that POST is healthy.
- **The improved error message also surfaces internal Firestore strings.** Acceptable here — admin-only endpoint, no PII in the messages — but worth a thought before applying the same pattern to user-facing routes.
- **PR #31's other items remain green.** This PR only touches the short-links index + POST catch; no other behavior changes.
