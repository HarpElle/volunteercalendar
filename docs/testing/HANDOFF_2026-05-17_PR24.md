## 1. Root causes (two unrelated, same symptom)

### Settings UI never reflected saved state
`GET /api/rooms/settings` returns `{ settings: {...} }` (consistent with the PUT response). The campus page did:

```ts
setSettings({ ...DEFAULT_ROOM_SETTINGS, ...data });
```

That spreads the **wrapper** object (with key `settings`) into the state, not the inner fields. Every named field in `DEFAULT_ROOM_SETTINGS` stayed at its default forever. Consequences:
- Checkbox always rendered unchecked on reload, regardless of DB state
- URL panel never appeared because `settings.public_calendar_token` was always `""`
- After save, the regenerated token was invisible until a full reload (and even then, see above — the checkbox would re-unmount)

### Every room had `public_visible: false`
- `POST /api/rooms` defaulted to `body.public_visible ?? false`
- Add/Edit Room forms never exposed a toggle
- `/api/calendar/public` correctly filters `data.public_visible !== false`

Net effect: zero rooms ever appeared on the public calendar, regardless of the org-wide flag. The token was valid, the API was healthy, the response was empty by construction.

## 2. Fixes

### Settings binding
- `setSettings({ ...DEFAULT_ROOM_SETTINGS, ...(data.settings || data) })` — unwrap correctly, falls back to `data` for old shape compatibility.
- `handleSave` now updates local state from the PUT response so a regenerated `public_calendar_token` shows up immediately without a reload.

### Public-visible toggle + default
- `POST /api/rooms`: `public_visible: body.public_visible ?? true` (was `?? false`).
- Add Room modal: new "Show on public calendar + iCal feed" checkbox (default on, resets on cancel).
- Edit Room form: same toggle. Existing rooms with `public_visible === undefined` render as checked — forward-compatible with the new default.
- Hint text under each toggle: "Untick for sensitive rooms (e.g. counseling). Has no effect if the org-wide public calendar is disabled."

### Backfill
- `scripts/backfill-rooms-public-visible.ts` (idempotent). Walks every church, flips rooms where `public_visible !== true`. Ran `--apply` in production:
  - 6 rooms in `TESTER — Codex 2` flipped from `false → true`
  - No other orgs had affected rooms

## 9. Edge cases worth watching

- **Default = visible.** New rooms now default to public-visible. Admins must explicitly opt sensitive rooms out. Flag if this surprises anyone.
- **`undefined` treated as visible.** The Edit Room form treats `room.public_visible === undefined` as `true` (checkbox renders checked). This matches the new POST default and makes legacy/imported rooms behave sensibly.
- **Token rotation on enable.** When you flip `public_calendar_enabled` from false → true, the server regenerates `public_calendar_token` (existing behavior). The campus page now refreshes from the PUT response so the new token is visible immediately.
- **Token rotation does NOT happen on save when the flag stays enabled.** Toggling other settings while enabled keeps the same token.
- **Per-room iCal feed (PR #23) is unaffected by `public_visible`.** The per-room iCal endpoint validates via the room's own `calendar_token` regardless of public_visible — that's intentional (subscribers paste a specific URL for a specific room).
- **Church-wide iCal feed (`/api/calendar/church/.../[token]`) honors `public_visible`.** Same as the new JSON public-calendar endpoint.
