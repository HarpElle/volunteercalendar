## 1. Root causes

### Per-room iCal 500
`/api/calendar/room/[roomId]/[calendarToken]` scans `collectionGroup("rooms").where("calendar_token", "==", token)` to find which church owns the token. That field had **no collection-group exemption** in `firestore.indexes.json`, so the query failed with `FAILED_PRECONDITION`. The catch block returned an opaque `"Server error"` body, hiding the cause from anyone reading the response.

### Public calendar 401 + "Missing church_id or token"
Three layered problems on the same surface:
- **Token-only URL** rendered "Missing church_id or token parameter" because the page required both query params and had no way to resolve `church_id` from a token alone.
- When both were supplied, the page called `/api/reservations?public_token=...` — but that endpoint has always required Bearer auth and has never read `public_token`. Returned 401. The page swallowed it as "Public calendar not available."
- There was no public, no-auth JSON endpoint backing the page.

### First-load UTC race
PR #22's room detail page kept `churchTimezone` state defaulting to `"UTC"` until the Firestore fetch resolved. The reservations fetch fired before TZ was known and briefly used UTC-today as `date_from`. PR #22 narrowed this to a one-frame flash; this PR eliminates it.

## 2. Fixes

### Firestore (already deployed and built)
- **`firestore.indexes.json`** — added single-field **collection-group exemptions** for `rooms.calendar_token` and `roomSettings.public_calendar_token`.

### New endpoint
- **`src/app/api/calendar/public/route.ts`** — public, no-auth JSON. Validates `token` against the church's `roomSettings.public_calendar_token` AND `public_calendar_enabled`. Accepts `(token + church_id)` or `(token)` alone (token-only resolves via the new `roomSettings` collection-group exemption). Hides rooms with `public_visible === false`. Excludes cancelled/denied. Returns `{ church, rooms, reservations, today }` where `today` is church-TZ.

### Rewrites
- **`/calendar/public` page** — calls the new endpoint, snaps the week view to the church's today on first load, navigates via TZ-safe date string arithmetic.
- **Per-room iCal endpoint** — catch block returns "Calendar feed is initializing" for `FAILED_PRECONDITION`. Defense in depth; the exemption fix prevents this from firing in normal operation.

### Small additions
- **Campus settings Public Calendar section** now shows BOTH the browser URL `/calendar/public?church_id=…&token=…` AND the iCal URL, with a note about `&embed=true`. Previously only the iCal URL was exposed and admins had to construct the HTML URL by hand.
- **Room detail page** — `churchTimezone` defaults to `null`; reservations fetch waits until it loads. No more UTC-today flash on first render.

## 9. Edge cases worth watching

- **iCal X-WR-TIMEZONE.** The feed already declares `X-WR-TIMEZONE: America/Chicago` (sample from production). Most calendar apps honor this. Flag any that don't.
- **Public calendar church-name title.** The page title now reads `"<Church Name> — Room Calendar"`. If the church doc has no name, falls back to `"Room Calendar"`.
- **Token-only URL discovery.** The token-only resolution does one collection-group lookup per request. Cheap; not cached. If multiple churches ever reuse the same token (shouldn't happen — tokens are 32 hex chars), the first match wins.
- **public_calendar_enabled toggle.** Flipping this off doesn't rotate the token. Re-enabling does (per the campus settings handler I left alone). Flag if you'd prefer disable-doesn't-rotate.
- **Room detail first-load.** The reservations fetch now waits for the church TZ. If you have a slow Firestore region the page may show a spinner slightly longer than before. Flag if it feels off.
