## 2. Root cause per issue

### Issue 1 — Service-plan date off by one day
**Symptom:** Modal picked `2026-05-24`, editor showed `Saturday, May 23, 2026`. Usage reports correctly recorded `May 24` because the storage was right.

**Cause:** `new Date("2026-05-24")` parses as **UTC midnight**. In any negative-offset timezone (every US zone), `.toLocaleDateString()` then renders the prior day. The plans editor and plans list both used this pattern.

**Fix:** New `formatLocalDate(iso, opts)` helper anchors the YYYY-MM-DD string to local noon (`new Date(${date}T12:00:00)`), which sits inside the same calendar day for every US timezone. Applied to plan editor header (line 335) and plans list `formatServiceDate` (line 18) + `isUpcoming` comparator.

### Issue 2 — ProPresenter export not discoverable
**Symptom:** No "Export for ProPresenter" button visible on the published plan, in the More menu, or anywhere else.

**Cause:** The API endpoint `/api/service-plans/[id]/export-propresenter` was fully implemented (works fine, returns a downloadable JSON), but the button was never wired into the plan editor UI.

**Fix:** Added an `Export for ProPresenter` button next to `Stage Sync` on the plan editor, rendered only when `isPublished && items.length > 0`. Clicking it calls the endpoint, receives the JSON blob, and triggers a browser download as `propresenter_<date>.json`. Errors render in a red banner above the metadata section.

### Issue 3 — Kiosk check-ins invisible to dashboard, reports, CSV, teacher room view, checkout
**Symptom:** Kiosk shows "Checked In!" with security code, but `/dashboard/checkin` counts stay at 0, daily and by-room reports show empty, CSV export is empty, and the teacher room view (if it loaded) would show no children.

**Cause:** Same unified-vs-legacy drift class as PR #10's household bug, but worse. The kiosk lookup endpoint (`/api/checkin/lookup`) returns **unified Person doc IDs** as `child_id` for Pro-tier orgs. Multiple downstream consumers then ran `collection("children").doc(child_id).get()` and silently `continue`d when the lookup failed.

Most damaging: **the kiosk check-in endpoint itself** (`/api/checkin/checkin`) silently skipped Person IDs, which means **no session was ever written to Firestore** for unified-mode orgs. The kiosk's "Checked In!" UI was lying — the security code was generated, but no session existed for the dashboard/report to display.

**Fix:** Created `src/lib/server/checkin-helpers.ts` with two helpers:

- `loadChild(churchRef, childId)` — checks `children` (legacy) first, then `people` with `person_type === "child"` (unified). Returns a `NormalizedChild` with consistent fields (first_name, last_name, preferred_name, display_name, has_alerts, allergies, medical_notes, default_room_id, grade) regardless of which collection it came from.
- `loadHouseholdPhone(churchRef, householdId)` — checks `checkin_households` (legacy) first; for unified `households` it looks up the first linked adult Person and returns their phone.

Applied to:
- Kiosk checkin loop — now writes sessions for unified-mode child IDs (this was the root cause of zero visibility)
- Kiosk SMS phone resolution — unified-aware
- Report `enrichSessions` — child names resolve in unified mode
- Report `first_time` case — reads both `checkin_households` and `households`
- Teacher room view — both child and phone unified-aware
- Admin checkout — child name unified-aware

### Issue 4 — Teacher Room URL fails with "Missing church_id or token parameter"
**Symptom:** Click `Copy Room View URL`, paste → page says missing param.

**Cause:** The copy handler built `${origin}/checkin/room/${room.id}?token=${token}`. The route requires both `church_id` AND `token` (rooms are stored under `churches/{cid}/rooms/{rid}`, so the route can't look up the room without the church scope).

**Fix:** Updated `copyRoomViewUrl` to include `&church_id=${churchId}` in the URL.
