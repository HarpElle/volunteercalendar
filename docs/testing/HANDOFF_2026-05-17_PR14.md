## 1. Root cause

Both `/dashboard/rooms` and `/dashboard/rooms/[id]` fetched the church's `subscription_tier` via `/api/church-info` (added in PR #13). That endpoint is intentionally **public** — it's used by unauthenticated join/invite pages and only exposes `id`, `name`, `type`, and `short_code`. It does **not** return `subscription_tier`.

So my fetch in PR #13 always saw `t.subscription_tier === undefined`. The tier state stayed at the `"free"` default. `TIER_LIMITS.free.rooms_recurring === false`. The booking wizard's step indicator, body, and Next-button logic all gate the recurrence step on the `recurringEnabled` prop, so step 4 was silently skipped on every tier — including Pro.

The wizard's recurrence logic itself was never broken; it just was never reached after PR #13.

## 2. Fix

Both pages now read the church doc directly from Firestore client-side, matching the established pattern in [src/app/calendar/page.tsx:25-31](src/app/calendar/page.tsx#L25). Firestore rules already allow active members to read their own church doc, so this is a one-call change with no new auth or API surface.

I did NOT add `subscription_tier` to `/api/church-info` because that endpoint is public and shouldn't expose org-wide product configuration to anonymous users hitting join links.

### Bundled minor improvements (also flagged in your feedback)
- Review step now shows `Organizer: <your displayName>`. The value was always being set server-side from `membership.display_name` (the actual stored `requested_by_name`), it was just never surfaced in the UI.
- Details step now shows a small helper line: "You'll be recorded as the organizer: \<name\>". Matches the test plan's organizer expectation without adding a redundant input.
- Review step's `Repeats` field is now a human-readable string like `Weekly, every Wednesday, for 12 occurrences` instead of a bare `Yes`. One-time bookings now show `Repeats: One-time booking` instead of being hidden.

## 9. Edge cases worth watching

- **Tier resolution timing.** Tier is now read from Firestore on mount. There's a brief moment between the page rendering and the tier coming back where `recurringEnabled` is `false`. If you click `Book this room` *very* fast, the form may open with the old default. The fix is to wait ~100ms before clicking, or refresh and try again. Note if you see this in practice.
- **Recurrence summary text.** I render the rule as "Weekly, every Wednesday, for 12 occurrences". Other valid combinations: "Every 2 weeks, every Wednesday, until 2026-08-12", "Monthly (by date), for 6 occurrences", etc. Flag if anything reads awkwardly.
- **Organizer attribution.** The helper line + Review row read from Firebase Auth `displayName`. If your account has no displayName set, it falls back to email, then to "current user" / "You". The server still uses `membership.display_name` for the actual stored `requested_by_name`, which may differ. Flag if you see a mismatch between what the wizard shows you and what the reservation list shows after submit.
