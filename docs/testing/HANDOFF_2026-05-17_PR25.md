## 1. Root cause / intended design

Three issues compounded:

1. **Free orgs couldn't even see the section.** `FacilitySharingSection` was nested inside `RoomsSettingsSection`, which only renders when `tier.rooms_enabled` (Starter+). A Free invitee had no Setup Code panel, no pending-invite list, no Accept button. Cross-org test was impossible.

2. **Test plan said "email," UI required a 6-letter Setup Code.** TEST_PLAN.md 5.9 read "Send an invite to the 2nd org by entering its admin email." Production `Invite Org` button opens a field labeled `Setup code` and validates via `/api/church-info?id=<code>`. Codex's email caused a 404. The UI's tiny "Setup code" placeholder was easy to miss because the surrounding button text said "Invite Org" — implying email.

3. **Dead tier logic.** The `tierLimits = TIER_LIMITS[church_id ? "growth" : "free"]` ternary always resolved to "growth" (because church_id is always truthy). So there was no visible "Upgrade" message and the host-create input was always shown — even on Free, where the underlying API would have allowed it but the test plan implied it shouldn't.

### Intended design (now consistent)
- **Hosting** a facility group → requires **Growth+** (matches the long-standing `TIER_LIMITS.facility_sharing` flag).
- **Invitees can be on any tier.** They share their Setup Code and click Accept. Matches the scenario ("outside school that rents Fellowship Hall").
- The Setup Code panel + pending-invite list + Accept button render on every tier so the cross-org flow works end-to-end.
