## Requirements for Unified Model

### Functional Requirements:
1. A **Person** can be any combination of: volunteer (scheduled for services), check-in guardian (picks up children), logged-in user (has Firebase Auth), or none of the above (just a contact record)
2. **Children** need check-in records, family linkage, medical info, and authorized pickup lists. Children are never volunteers.
3. **Households** link adults and children. Two parents may both volunteer on different teams but share the same children. A single parent with 3 kids is one household. A married couple where only one volunteers is still one household.
4. The **scheduling algorithm** must still be able to query "all active volunteers in ministry X with their blackout dates and max frequency" in a single Firestore query
5. The **check-in kiosk** must still be able to search by phone number and return the full family (adults + children) efficiently
6. **Admin views** need to list all people with filters for: role, ministry, status, type (adult/child), household

### Technical Constraints:
- Firestore — no JOINs, no server-side aggregation beyond count()
- All timestamps as ISO strings (not Firestore Timestamps, not JS Date objects)
- Collection names in kebab-case
- Document IDs are auto-generated (Firestore default)
- Must work with Firebase v9+ modular SDK (not the compat layer)
- Client-side reads should minimize document count (Firestore charges per read)
- The existing scheduling algorithm is 3-phase: Greedy Assignment -> Conflict Detection -> Local Search for Fairness. It currently takes a `Volunteer[]` array. The new model must produce a compatible data structure.

### Permission Model Requirements:
- 4 roles: owner > admin > scheduler > volunteer (keep these, do NOT add new roles)
- Permission flags on Membership: `event_coordinator`, `facility_coordinator`, `checkin_volunteer`
- Schedulers have `ministry_scope: string[]` to limit which teams they can schedule
- Team Leader = a scheduler with `ministry_scope` set to their team(s)
- Admin and Owner implicitly have all permission flags
- Need a `canPerformAction(membership, action, resourceId?)` utility function pattern

## Anti-Patterns to Avoid

Please explicitly address why you did or did not choose these patterns, since they're tempting but often wrong for Firestore:

1. **Deep nesting:** `churches/{cid}/households/{hid}/people/{pid}/scheduling-profile/{spid}` — too many subcollections means too many reads
2. **Single mega-document:** Putting everything (scheduling, check-in, household, children) in one Person document that exceeds 1MB
3. **Excessive denormalization:** Copying volunteer names into every Assignment document (stale data risk)
4. **Collection group queries everywhere:** They work but have limitations (can't combine with certain filters)
5. **Relying on client-side joins:** Fetching People then fetching Households then merging — too many round trips
